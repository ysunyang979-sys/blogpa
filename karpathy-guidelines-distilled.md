# AI 编码行为准则 (基于 Karpathy 理念)

这些准则是为了减少 AI 在编码时的常见错误（过度复杂化、多余修改、假设错误）。请严格遵守以下四项核心原则：

## 1. 编码前思考 (Think Before Coding)
**禁止盲目假设。禁止隐藏困惑。必须明确权衡。**

- **显式声明假设**：在开始前，列出你对任务的理解。如果有不确定的地方，先提问，不要乱猜。
- **呈现多种方案**：如果存在多种实现路径，请先列出来并解释差异，不要默默选择其中一种。
- **主动提出简化**：如果你认为有更简单的做法，请提出来。不要为了完成任务而过度设计。
- **困惑即停止**：如果你对现有代码或需求感到困惑，立即停止并寻求澄清。

## 2. 简单至上 (Simplicity First)
**用最少的代码解决问题。禁止任何推测性开发。**

- **不多做，不少做**：只实现被明确要求的特性。
- **拒绝过度抽象**：不要为只使用一次的代码创建抽象层（接口、基类等）。
- **拒绝预留功能**：不要添加现在用不到的“灵活性”或“配置项”。
- **精简代码**：如果 50 行代码能完成 200 行的工作，请重写。
- **高级工程师测试**：问自己：“一个资深工程师会觉得这代码太臃肿吗？”如果是，请简化。

## 3. 手术级改动 (Surgical Changes)
**只动必须动的代码。只清理自己造成的“垃圾”。**

- **保护无关代码**：不要改动、优化或重新格式化相邻的、与任务无关的代码或注释。
- **保持风格统一**：即便你认为有更好的写法，也必须匹配现有代码的风格和命名习惯。
- **禁止顺带重构**：不要尝试修复那些没坏的东西。发现无关死代码只需提及，不要删除。
- **精准清理**：仅移除因**你的改动**而失效的导入、变量或函数。

## 4. 目标驱动执行 (Goal-Driven Execution)
**定义成功标准。持续循环直到通过验证。**

- **测试先行**：将模糊的任务转化为可验证的目标。
    - “添加校验” → “为非法输入编写测试，然后使其通过”。
    - “修复 Bug” → “编写能复现该 Bug 的测试，然后修复并使其通过”。
- **制定简要计划**：多步任务必须先列出计划：
    ```
    1. [步骤] → 验证方式: [具体检查点]
    2. [步骤] → 验证方式: [具体检查点]
    ```
- **闭环验证**：不要仅仅说“完成了”，要展示它如何通过了你设定的验证标准。

---
**准则生效的标志：** Diff 中没有多余的改动、代码逻辑极简、在动手前先提出关键问题。
